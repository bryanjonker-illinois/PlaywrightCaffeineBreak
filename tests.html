Where does Playwright fit into this testing mindset?

Playwright, in my mind, has two main purposes:

  1. It allows unit testing on web pages when these components aren't part of a server-side suite. PHP, .NET, Java all have their unit test framework, but you can't run their unit tests on pure JavaScript.
  2. Overall smoke tests and monkey tests on existing websites. It doesn't matter what the website was coded in, Playwright can test it.

We will be looking at the first method here, but I feel like both options are equally important.

Can Bryan get on his soapbox on xxxxxxx-driven development?

Of course he can!

Test-driven development was explained to me as "you write tests first," but I feel it is more than that. We all do some type of xxxxx-driven development, with the xxxxx being what we care about. How we code is literally driven by that X. Do we care that it works? Do we care that it can be tested? Do we care that it is accessible? Do we care that it is easy to use? What xxxxx-driven development does is it names what we care about, and it says that when we build a project, we are focusing on that piece.

Follow up question. Is it OK not to care?

Of course. We all are busy and not all projects matter equally. Obligatory XKCD comic. Some things we will spend a lot of time on and it will never get used. Others will have a life of their own. We do our best.
